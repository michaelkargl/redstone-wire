## Implementation Notes

### Texture reuse strategy
Initially all new blocks reuse the existing `redstone_chain` and `redstone_chain_powered` textures. This means all 4 block types look identical in-game. Custom textures can be added in a follow-up task. The models reference textures by path, so swapping textures later only requires updating the model JSON files.

### Connector blockstate — empty string key
When a block has NO blockstate properties, the blockstate JSON uses an empty string `""` as the variant key. This is standard Minecraft convention. Example: `"variants": { "": { "model": "..." } }`.

### Item model patterns
The existing `redstone_chain.json` item model uses `minecraft:item/generated` with a separate layer0 texture (the chain texture). For the new blocks, using `parent: "redstone_wire:block/redstone_input"` (block model reference) is simpler and shows the block's 3D shape in inventory, which is more standard for block items.

### File count summary
- 3 blockstate files (redstone_input.json, redstone_output.json, redstone_connector.json)
- 5 block models (input, input_powered, output, output_powered, connector)
- 3 item models (input, output, connector)
- 1 modified file (en_us.json — 3 new entries)
- **Total: 11 new files, 1 modified file**

### Existing textures referenced
The texture files themselves live at:
- `src/main/resources/assets/redstone_wire/textures/block/redstone_chain.png`
- `src/main/resources/assets/redstone_wire/textures/block/redstone_chain_powered.png`

These do NOT need to be copied or modified — the new models reference them by path.
